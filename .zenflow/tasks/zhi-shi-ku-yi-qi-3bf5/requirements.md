# knowledge-indexer 需求分析

## 1. 需求背景与目标

### 1.1 背景
- 当前已有一个 Java 后端项目，已经承担对外 API、用户体系、鉴权、权限控制和业务聚合能力。
- 现阶段需要建设企业知识库一期能力，但不希望把索引构建、增量同步、向量检索等 AI 数据处理能力硬塞进现有 Java 项目，避免增加维护复杂度。
- 需求方已明确倾向于新建独立项目 `knowledge-indexer`，定位为知识处理、索引构建、内部检索能力服务，不包含前端页面，也不替代现有 Java 业务主后端。

### 1.2 目标
- 建立一个独立的知识索引服务，统一接入文件、数据库、内部/外部 API 等多种知识来源。
- 将多源知识加工为统一的文档、切片、向量和元数据索引，支持后续检索和问答。
- 支持全量同步与增量同步，确保知识索引可持续更新、可追溯、可运维。
- 通过内部 API 向现有 Java 后端提供搜索、问答、任务状态、数据源管理等能力。

### 1.3 一期范围
- 新建独立项目：`knowledge-indexer`
- 项目职责：
  - 数据源接入
  - 文本抽取、清洗、切分
  - embedding 生成
  - 索引写入与更新
  - 内部检索 API
  - 内部问答 API
  - 同步任务管理与健康检查
- 一期优先支持的数据源：文件源、API 源；数据库源作为紧随其后的扩展能力。
- 一期不包含：
  - 前端页面或管理后台
  - 对外开放接口
  - 用户中心、统一登录、权限中心
  - Neo4j 图谱、复杂重排平台等二期增强能力

### 1.4 目标用户
- 直接使用方：现有 Java 后端服务、后端研发同学、平台运维同学。
- 间接受益方：最终使用知识检索/问答能力的业务系统与业务用户。

## 2. 功能清单

### 2.1 主流程
1. 管理员或系统配置知识数据源。
2. 触发一次全量或增量同步任务。
3. 系统从数据源拉取原始数据并做标准化处理。
4. 系统执行知识处理流程：抽取文本、清洗内容、切分 chunk、生成 embedding。
5. 系统将文档、chunk、向量、任务状态等写入索引存储。
6. Java 后端通过内部 API 调用检索/问答能力。
7. `knowledge-indexer` 根据查询条件、权限上下文返回候选片段、引用和任务结果。

### 2.2 核心功能

#### 2.2.1 数据源管理
- 支持新增、查看和维护数据源配置。
- 支持至少三类来源的统一抽象：
  - 文件源：本地目录、挂载目录、对象存储同步目录
  - 数据库源：业务表、FAQ 表、配置表
  - API 源：内部知识接口、业务系统接口、第三方聚合接口
- 支持为数据源定义唯一标识、来源类型、拉取方式、同步策略和增量游标。

#### 2.2.2 索引构建与同步
- 支持手动触发同步任务。
- 支持全量同步与增量同步两种模式。
- 支持记录同步任务状态、开始/结束时间、失败原因、重试情况。
- 支持索引重建和断点续跑所需的基础任务信息管理。

#### 2.2.3 文档处理
- 支持原始内容抽取和文本清洗。
- 支持将文档切分为可检索的 chunk。
- 支持生成摘要、标签等可扩展的结构化信息。
- 支持保留文档到 chunk 的追溯关系，便于后续引用返回。

#### 2.2.4 检索能力
- 提供内部搜索接口。
- 支持基于向量的相似检索。
- 支持基于元数据的过滤能力，例如数据源、文档类型、业务字段等。
- 支持在检索阶段接收 Java 传入的权限上下文，并进行 ACL 过滤。
- 返回结果需包含候选片段、相关文档信息和引用依据。

#### 2.2.5 问答能力
- 提供内部问答接口。
- 基于检索结果进行答案组织，并返回引用片段。
- 当证据不足时，能够返回兜底提示，而不是编造结论。

#### 2.2.6 运维与可观测性
- 提供健康检查接口。
- 提供任务状态查询接口。
- 支持记录失败样本、同步快照或必要日志，便于排障。

### 2.3 边界场景

#### 2.3.1 数据源侧边界
- 文件为空、内容损坏、编码异常时，任务需记录失败原因，不应导致整批任务完全不可见。
- API 源超时、限流或返回结构变化时，应记录错误并支持后续重试。
- 数据库源出现脏数据、主键缺失或增量字段异常时，应能够识别并上报。

#### 2.3.2 索引构建边界
- 文档内容过长时，应能够按规则切 chunk，而不是整篇失败。
- 文档重复导入时，应具备幂等处理或可识别更新逻辑。
- embedding 生成失败时，应对失败记录进行标记，避免无声丢失。

#### 2.3.3 检索问答边界
- 查询无命中时，应返回空结果或明确提示，而不是返回无关内容。
- 用户权限不足时，应过滤无权访问文档，不能直接泄露结果内容。
- 当检索结果质量不足以支持问答时，应返回“证据不足”类提示。

#### 2.3.4 系统边界
- `knowledge-indexer` 仅提供内部服务，不负责外部用户直接访问。
- Java 后端负责统一鉴权和业务入口，`knowledge-indexer` 仅消费其传入的权限上下文。
- 一期不要求建设前端页面；如后续有运营管理诉求，再单独补管理台。

## 3. 验收标准

### 3.1 业务范围验收
- 已明确项目边界：`knowledge-indexer` 是独立知识索引服务，不替代现有 Java 主后端。
- 已明确一期范围：聚焦索引构建、检索、问答、任务管理、健康检查，不包含前端。
- 已明确系统分工：Java 负责对外接口与权限体系，`knowledge-indexer` 负责知识处理与内部服务。

### 3.2 功能能力验收
- 能描述并覆盖文件源、API 源两类一期核心数据源，数据库源纳入后续扩展范围。
- 能支持新增数据源、触发同步、查询任务、执行检索、执行问答等核心能力。
- 能支持全量与增量同步两种索引更新方式。
- 检索结果具备文档/片段引用信息，并支持权限过滤。
- 问答结果具备引用依据，且证据不足时有兜底策略。

### 3.3 非功能验收
- 具备可观测的任务状态与健康检查能力。
- 具备失败记录、错误可追踪和后续重试的基本设计要求。
- 数据处理链路可追溯到数据源、文档、chunk 和同步任务层面。

## 4. 疑问点与待确认项

### 4.1 业务与权限
- Java 传给 `knowledge-indexer` 的权限上下文字段标准尚未最终明确，例如角色、部门、人员、资源标签的具体结构。
- 文档 ACL 是在索引阶段固化，还是在检索阶段动态映射，仍需业务侧进一步确认。

### 4.2 数据源接入
- 一期 API 源的具体来源系统清单、调用频率、鉴权方式尚未明确。
- 文件源是否需要直接对接 MinIO，还是先通过同步目录落地，再由索引服务处理，仍需确认。
- 数据库源虽然已列入推荐能力，但是否纳入一期交付范围仍需排期确认。

### 4.3 问答能力
- 一期问答接口是否直接接入大模型，以及采用何种模型/供应商，当前描述中尚未明确。
- 问答结果的输出格式要求（纯文本、结构化引用、置信度等）仍需补充。

### 4.4 运维与交付
- 任务调度由 `knowledge-indexer` 内部负责，还是由外部调度平台/Java 后端触发，尚未最终确认。
- 一期对性能指标的要求（索引耗时、检索延迟、数据规模、并发量）尚未给出明确基线。
- 部署环境虽然倾向于 `Docker Compose` 起步，但实际生产环境接入规范仍需运维侧确认。

## 5. 本阶段结论
- 本需求的本质不是建设一个完整知识库产品，而是建设一个可独立演进的“知识索引与检索底座”。
- 一期核心目标是先把知识接入、索引构建、检索问答、任务状态这条链路打通。
- 前端、复杂图谱、高级重排、完整运营后台等能力应作为后续阶段演进，而非一期阻塞项。
