# knowledge-indexer 二期需求分析

## 1. 需求背景与目标

### 1.1 背景
- 当前仓库已经交付 `knowledge-indexer` 一期骨架，具备文件源 / API 源接入、同步任务、文本清洗切分、内存索引、内部检索与问答接口等基础能力。
- 一期实现明显偏向“跑通链路”，默认依赖内存仓储与哈希向量，适合本地验证和接口联调，但不适合真实业务场景的稳定运行。
- 现有文档已经明确了一批二期预留点：PostgreSQL + pgvector 持久化、Redis 异步任务队列、MinIO 原文归档、CocoIndex 正式 Flow、`postgres` 数据源同步、真实 embedding / rerank 等。
- 当前任务“完善知识库二期的内容”结合仓库现状判断，不是重新定义一个全新产品，而是基于一期骨架，把二期应交付的能力边界、业务目标、关键场景和验收口径补充完整，为后续技术方案、任务拆分和实现提供统一依据。

### 1.2 目标
- 将知识库从“一期可演示原型”升级为“二期可持续接入、可持久化、可异步执行、可扩展”的内部知识基础设施。
- 明确二期范围内的关键能力：稳定的数据接入、可观测的同步流程、可持久化的索引存储、更可靠的检索与问答效果、更加清晰的运行边界。
- 在不破坏一期对外接口定位的前提下，补齐二期内容说明，使研发、联调、运维和接入方对系统职责和后续演进方向达成一致。
- 为后续输出 `spec.md`、`tasks.md` 提供可执行的业务输入，而不是仅停留在“建议演进方向”的泛化描述。

### 1.3 目标用户
- 后端研发：负责 Java 业务主后端与 `knowledge-indexer` 的内部联调。
- AI / 平台研发：负责索引、检索、问答、数据接入和基础设施接入。
- 运维 / SRE：负责环境部署、依赖服务配置、运行状态巡检。
- 业务接入方：负责提交数据源、触发同步、消费检索 / 问答结果。

## 2. 功能清单

### 2.1 主流程能力

#### 2.1.1 数据源接入能力完善
- 保留一期已有的 `file`、`api`、`postgres` 三类数据源模型。
- 二期需要把 `postgres` 从“占位类型”补充为“可用类型”，至少明确：
  - 连接配置结构
  - 支持的抽取字段 / 主键 / 增量列
  - 全量 / 增量同步规则
  - 错误处理与重试策略
- 对 `file`、`api` 数据源补充更清晰的内容约束和运行说明，例如文件格式支持范围、API 超时 / 分页 / 限流说明等。

#### 2.1.2 同步任务从串行内存执行升级为可追踪异步执行
- 一期 `SYNC_RUN_INLINE` 适合本地验证；二期需要明确支持异步同步任务。
- 同步任务应具备：排队、执行中、成功、失败、取消 / 中断（如暂不实现需在边界中说明）等状态语义。
- 任务结果需要可追踪，至少能定位：处理文档数、失败文档数、失败原因摘要、开始 / 完成时间。
- 需要说明全量同步、增量同步、重建索引的使用场景与优先级。

#### 2.1.3 索引与存储能力完善
- 二期需要把当前内存仓储替换或抽象为持久化存储，保证服务重启后数据不丢失。
- 需要明确文档、切片、向量、checkpoint、任务状态等核心数据在持久化层的存储要求。
- 需要说明向量索引的生产化能力目标，包括真实 embedding、向量检索能力、后续 rerank 预留。
- 需要补充原文与失败样本归档诉求，便于排障与复盘。

#### 2.1.4 检索与问答内容完善
- 检索接口继续作为 Java 后端的内部能力，不直接暴露终端用户体系。
- 二期要明确检索结果质量提升方向：
  - 更稳定的召回能力
  - 支持更丰富的过滤条件
  - 保持 ACL 过滤能力
  - 为 rerank / 多路召回预留扩展点
- 问答能力继续基于“检索优先、证据驱动”的模式，至少需要明确：
  - 证据不足时的兜底策略
  - 引用片段和来源信息的返回要求
  - 与真实大模型 / embedding 模型对接后的边界说明

#### 2.1.5 运维与联调内容完善
- 需要补齐二期部署与依赖说明，包括 PostgreSQL、Redis、MinIO 等基础设施在系统中的职责。
- 需要明确配置项、健康检查口径、关键日志 / 指标范围、常见联调步骤。
- 需要帮助接入方理解“哪些问题由 Java 后端负责，哪些问题由 knowledge-indexer 负责”，避免职责混淆。

### 2.2 边界场景 / 异常场景

#### 2.2.1 数据源配置异常
- 文件目录不存在、路径无权限、匹配不到文件。
- API 源未配置 `base_url`、鉴权失败、超时、分页异常。
- Postgres 源连接失败、表 / 字段不存在、增量列不可用。
- 不同数据源配置合法性校验失败时，需要给出明确错误信息。

#### 2.2.2 同步执行异常
- 单篇文档解析失败时，是否允许任务整体继续。
- 网络波动、上游限流、临时超时后的重试规则。
- 重复触发同步时的幂等策略和并发限制。
- 长时间运行任务的可观测性与状态更新频率。

#### 2.2.3 检索 / 问答异常
- 索引为空或尚未完成同步时的提示。
- 检索结果命中不足时的兜底文案与返回结构。
- ACL 过滤后无结果时的表现形式。
- 模型不可用、向量生成失败、rerank 未启用时的退化策略。

#### 2.2.4 基础设施边界
- PostgreSQL、Redis、MinIO 配置存在但服务不可用时，健康检查如何表达。
- 本地开发模式与生产模式能力差异需要清晰标注，避免接入方误以为某些能力已经正式可用。
- 若部分二期能力仍保留占位，需要在文档中明确“可配置但未正式启用”的范围。

## 3. 非功能性要求

### 3.1 稳定性
- 服务重启后，数据源、任务记录、索引数据不应因默认实现而整体丢失。
- 同步任务失败后，应能通过任务记录定位原因，而不是仅依赖控制台日志。

### 3.2 可维护性
- 二期内容必须延续一期的模块边界，避免将 Java 业务逻辑重新耦合进 `knowledge-indexer`。
- 接口语义尽量兼容一期，降低接入改造成本。
- 文档应明确“当前已实现 / 后续预留 / 暂不支持”三类状态，减少误解。

### 3.3 可扩展性
- 数据源、索引流程、向量模型、重排能力应保留替换点。
- 数据结构与接口设计要允许未来扩展更多数据源、更多检索策略和更复杂的权限维度。

### 3.4 安全性
- 继续采用内部接口访问模型，避免对外暴露内部数据面。
- 保持并强化 ACL 过滤、内部 Token 鉴权、敏感配置隔离等能力。
- 对数据库 / API 连接配置中的敏感信息，需要明确脱敏展示与日志边界。

## 4. 验收标准

### 4.1 需求内容层面的验收
- 能清楚说明一期现状与二期目标的关系，避免“二期只是一些技术建议”的模糊表述。
- 能明确列出二期必须补充的核心能力：持久化、异步任务、生产级数据源、检索 / 问答增强、运维联调说明。
- 能说明每类能力的主流程与关键异常场景，而不是仅罗列名词。
- 能体现系统边界：Java 后端仍负责用户体系与外部业务接口，`knowledge-indexer` 负责内部知识处理与检索能力。

### 4.2 交付物层面的验收
- 输出 `requirements.md`，覆盖“需求背景与目标 / 功能清单 / 验收标准 / 疑问点”四个部分。
- 文档内容应与当前仓库实现保持一致，不得把仓库尚无迹象支持的能力写成既有事实。
- 文档中需显式标注基于仓库现状做出的合理推断，供后续技术方案阶段进一步确认。

### 4.3 面向后续阶段的可执行性验收
- 后续编写技术方案时，可以直接据此展开模块设计、接口定义和工作量评估。
- 后续拆分任务时，可以围绕明确能力项进行拆解，而不是重新回到“二期到底做什么”的讨论。

## 5. 疑问点 / 待确认事项
- “完善知识库二期的内容”中的“内容”是否主要指：
  - 二期产品 / 技术能力范围的补充说明；
  - 还是特指某一批知识库文档、FAQ、使用手册等内容资产本身的补写。
- 二期是否要求真正引入大模型生成答案，还是仍以检索拼装为主，仅对 embedding / rerank 做增强。
- `postgres` 数据源在二期的优先级是否高于其他增强项；若资源有限，是否允许先完成持久化与异步任务，再补数据库源。
- Redis / MinIO / pgvector / CocoIndex 在本阶段是“必须落地实现”，还是允许先完成接口与文档约束、后续分批接入。
- 二期交付的首要目标是“生产可用”还是“联调可用 + 架构预埋更完整”；该判断会直接影响验收口径。
- 是否需要新增对外说明文档，用于指导 Java 接入方或业务团队完成二期接入与迁移。

## 6. 需求分析结论
- 基于当前仓库上下文，本次任务更合理的解读是：围绕 `knowledge-indexer` 一期骨架，补齐知识库二期的能力范围说明和交付边界，使系统从原型阶段过渡到可持续演进的内部知识基础设施。
- 二期关注点不在“新增一个完全不同的业务”，而在“把一期预留能力从占位和建议，收敛为清晰、可实现、可验收的内容定义”。
- 后续技术方案阶段应优先围绕持久化、异步任务、`postgres` 数据源、生产级检索 / 问答、运行与联调规范五条主线展开。
