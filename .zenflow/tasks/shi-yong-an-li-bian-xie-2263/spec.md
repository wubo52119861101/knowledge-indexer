# 使用案例编写 - 技术方案

## 1. 方案目标

### 1.1 背景判断
当前仓库已经具备两类说明材料：
- `README.md`：偏项目简介与快速启动；
- `docs/usage.md`：偏接口说明、启动步骤与联调约束。

但需求目标不是继续补充一份 API Reference，而是沉淀一份“面向使用者的场景化使用案例/使用手册”。因此本方案聚焦“文档产品化改造”，而非系统功能开发。

### 1.2 本次技术方案目标
后续实现阶段需要产出一套可维护、与现有代码一致、便于首次接触者快速上手的使用案例文档体系，重点解决以下问题：
- 把现有“接口说明”转化为“按场景操作”的表达方式；
- 明确一期真实可用能力与占位能力，避免误导；
- 让接入开发、联调测试、项目负责人都能快速定位适合自己的阅读入口；
- 建立文档内容与代码接口之间的映射关系，降低后续维护成本。

### 1.3 交付策略
建议后续落地采用“新增主文档 + 保留现有参考文档”的方案：

1. 新增 `docs/use-cases.md` 作为主使用手册；
2. 保留 `docs/usage.md` 作为偏接口/联调参考文档；
3. 后续如需增强入口，可在 `README.md` 增加“使用手册”链接。

选择该方案的原因：
- `docs/usage.md` 已具备较强参考手册属性，直接大改会打断现有阅读路径；
- 新增独立文档更利于围绕“角色 + 场景 + 步骤 + 结果”重组内容；
- 参考文档与场景文档分层后，职责更清晰，也更利于后续维护。

## 2. 整体架构设计

### 2.1 文档信息架构
后续文档产物建议采用如下结构：

1. **项目定位与适用场景**
   - 项目解决什么问题
   - 当前适合谁使用
   - 一期适用范围与不适用范围

2. **快速开始**
   - 环境要求
   - `.env` 配置
   - 启动方式
   - `/`、`/health`、`/docs` 验证方法

3. **典型使用案例**
   - 案例 1：本地启动并确认服务可用
   - 案例 2：创建文件数据源并完成全量同步
   - 案例 3：创建 API 数据源并完成增量同步
   - 案例 4：执行检索验证
   - 案例 5：执行问答验证
   - 案例 6：带 ACL 上下文联调

4. **标准操作流程**
   - 创建数据源
   - 触发同步
   - 查看任务状态
   - 搜索验证
   - 问答验证

5. **异常与边界说明**
   - Token 未配置/错误
   - `SYNC_RUN_INLINE=false`
   - 内存仓储导致重启丢数据
   - `postgres` 仅占位
   - 证据不足兜底
   - 健康检查仅代表配置状态

6. **FAQ / 注意事项**
   - 如何判断是否索引成功
   - 为什么健康检查通过但业务不可用
   - 为什么问答没有“大模型感”
   - 为什么重启后数据消失

### 2.2 文档分层设计
为避免内容混杂，建议采用“两层文档”模型：

| 层级 | 文档 | 角色 | 目的 |
| --- | --- | --- | --- |
| L1 | `README.md` | 所有读者 | 提供项目入口与跳转 |
| L2-A | `docs/use-cases.md` | 首次使用者、联调、测试 | 提供场景化操作手册 |
| L2-B | `docs/usage.md` | 开发、接入方 | 提供接口与配置参考 |

### 2.3 内容生产架构
后续文档编写应基于“代码事实源 + 现有文档参考源”生成，而不是凭经验补写。

```text
代码实现（app/api, app/services, app/schemas）
        +
现有说明（README.md, docs/usage.md）
        ↓
能力核对与边界确认
        ↓
场景拆解（目标/前置条件/步骤/结果/注意事项）
        ↓
docs/use-cases.md
```

### 2.4 章节编排原则
- 先讲“适合谁、解决什么问题”，再讲“怎么操作”；
- 先给完整成功路径，再补异常与边界；
- 每个案例都必须包含“目标、前提、步骤、预期结果、常见失败提示”；
- 所有能力描述均以当前仓库已实现逻辑为准，不扩写二期能力。

## 3. 接口 / 数据结构定义

本项目的“技术方案”虽然服务于文档编写，但仍需明确后续手册中要引用的关键接口与数据结构，以保证编写内容可核对、可追溯。

### 3.1 关键接口清单

| 接口 | 方法 | 作用 | 代码位置 |
| --- | --- | --- | --- |
| `/` | `GET` | 根路径确认服务已启动 | `app/main.py` |
| `/health` | `GET` | 查看应用与依赖配置级健康状态 | `app/api/health.py` |
| `/internal/sources` | `POST` | 创建数据源 | `app/api/sources.py` |
| `/internal/sources/{source_id}` | `GET` | 查看数据源详情与最新任务 | `app/api/sources.py` |
| `/internal/sources/{source_id}/sync` | `POST` | 触发同步任务 | `app/api/sources.py` |
| `/internal/jobs/{job_id}` | `GET` | 查询同步任务结果 | `app/api/jobs.py` |
| `/internal/search` | `POST` | 检索验证 | `app/api/internal_search.py` |
| `/internal/ask` | `POST` | 问答验证 | `app/api/internal_ask.py` |

### 3.2 统一响应结构
除异常 HTTP 状态外，成功响应统一采用以下包裹结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_xxxxxxxxxxxx"
}
```

来源：`app/api/responses.py` 与 `app/main.py` 中的请求上下文中间件。

文档编写要求：
- 所有案例中的“预期结果”都应同时描述 HTTP 状态与 `data` 关键字段；
- 需要明确 `X-Request-Id` 可透传，便于联调排查。

### 3.3 鉴权约定

| 项 | 说明 |
| --- | --- |
| 鉴权头 | `X-Internal-Token` |
| 生效范围 | `/internal/*` |
| 校验逻辑 | 当 `INTERNAL_API_TOKEN` 为空时不校验；不为空时必须严格匹配 |
| 非鉴权接口 | `/`、`/health` |

来源：`app/core/security.py`。

### 3.4 关键请求结构

#### 3.4.1 创建数据源 `CreateSourceRequest`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `name` | `str` | 数据源名称 |
| `type` | `file/api/postgres` | 数据源类型 |
| `config` | `dict` | 类型相关配置 |
| `sync_mode` | `full/incremental/rebuild` | 默认同步模式 |
| `schedule` | `str \| null` | 预留字段 |
| `enabled` | `bool` | 是否启用 |

其中：
- `file` 类型至少要求 `config.root_path`；
- `api` 类型至少要求 `config.base_url`；
- `postgres` 当前仅允许创建，不支持成功同步。

来源：`app/schemas/source.py`、`app/services/source_service.py`。

#### 3.4.2 触发同步 `TriggerSyncRequest`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `mode` | `full/incremental/rebuild` | 本次同步模式 |
| `operator` | `str` | 操作人 |
| `options.force_rebuild` | `bool` | 预留选项，当前未实际驱动特殊流程 |

来源：`app/schemas/source.py`。

#### 3.4.3 检索请求 `SearchRequest`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `query` | `str` | 查询词 |
| `top_k` | `int` | 返回条数，`1~20` |
| `filters.source_ids` | `list[str]` | 按数据源过滤 |
| `filters.doc_types` | `list[str]` | 按文档类型过滤 |
| `filters.metadata` | `dict` | 按文档元数据精确匹配 |
| `acl_context.user_id` | `str \| null` | 用户身份 |
| `acl_context.roles` | `list[str]` | 角色集合 |
| `acl_context.departments` | `list[str]` | 部门集合 |
| `acl_context.tags` | `list[str]` | 标签集合 |

来源：`app/schemas/retrieval.py`。

#### 3.4.4 问答请求 `AskRequest`
结构与检索请求一致，仅主字段由 `query` 改为 `question`。

#### 3.4.5 任务结果 `JobItem`

| 字段 | 说明 |
| --- | --- |
| `id` | 任务 ID |
| `source_id` | 对应数据源 |
| `mode` | 同步模式 |
| `status` | `PENDING/RUNNING/SUCCEEDED/FAILED` |
| `processed_count` | 成功处理条数 |
| `failed_count` | 失败条数 |
| `error_summary` | 失败摘要 |
| `started_at` / `finished_at` | 执行时间 |

文档编写时应把 `SUCCEEDED`、`FAILED`、`PENDING` 作为案例成功/异常判断的重要依据。

### 3.5 使用案例的数据模板
为了保证后续每个案例写法统一，建议采用统一模板：

```markdown
## 案例 X：{案例标题}

### 场景目标
### 适用角色
### 前置条件
### 操作步骤
### 预期结果
### 常见问题 / 注意事项
```

补充约束：
- “操作步骤”中需要嵌入真实接口路径与示例请求；
- “预期结果”至少描述一个成功判断字段；
- “注意事项”必须突出一期限制或常见误解。

## 4. 核心逻辑说明

### 4.1 文档编写主流程
后续实现步骤建议按如下顺序开展：

1. **能力对齐**
   - 以 `app/api/*`、`app/services/*`、`app/schemas/*` 为事实源，确认接口、字段、行为；
   - 以 `README.md`、`docs/usage.md` 为补充说明源，吸收已有可复用内容。

2. **场景抽象**
   - 从需求分析中确认 6 个主流程案例与 6 个边界案例；
   - 将接口能力重组为“用户要完成什么事”的视角，而不是“接口怎么定义”的视角。

3. **案例落文**
   - 每个主流程案例编写：目标、前提、步骤、预期结果、注意事项；
   - 每个边界案例编写：触发条件、现象、原因解释、排查建议。

4. **一致性校验**
   - 逐项检查接口路径、字段名、枚举值、环境变量名；
   - 校验所有“支持/不支持”的结论是否与代码一致；
   - 避免引入仓库中不存在的脚本、页面或能力。

### 4.2 案例内容生成逻辑
每个案例都应遵循同一条叙事链：

```text
为什么做 → 做之前要准备什么 → 具体怎么做 → 成功时会看到什么 → 失败时如何理解
```

这条链路用于解决当前 `docs/usage.md` 偏“说明式”而非“操作式”的问题。

### 4.3 与代码行为的映射关系

#### 4.3.1 同步流程映射
后续文档应准确反映以下逻辑：
- 创建数据源后，通过 `/internal/sources/{source_id}/sync` 触发同步；
- 系统先创建 `job`；
- 当 `SYNC_RUN_INLINE=true` 时，任务会立即在当前进程内执行；
- 当 `SYNC_RUN_INLINE=false` 时，仅创建 `PENDING` 任务，不会自动消费；
- 成功后通过 `/internal/jobs/{job_id}` 查看处理结果。

来源：`app/core/container.py`、`app/services/job_service.py`、`app/services/indexing_service.py`。

#### 4.3.2 检索流程映射
后续文档应准确反映以下逻辑：
- 查询词先被转换为哈希向量；
- 与所有 chunk 向量计算相似度；
- 先执行过滤条件，再执行 ACL 命中判断；
- `deny` 优先于 `allow`；
- 最终按分数降序截取 `top_k`。

来源：`app/services/retrieval_service.py`。

#### 4.3.3 问答流程映射
后续文档应明确以下事实：
- `/internal/ask` 本质上先调用检索；
- 若命中文档数不足或首条分数低于阈值，则返回“证据不足”；
- 若证据充足，则把前若干检索结果拼接成答案；
- 当前不是大模型自由生成问答。

来源：`app/services/qa_service.py`。

### 4.4 边界说明生成逻辑
后续边界章节应按“现象 → 原因 → 结论”表达，避免只描述错误不解释原因。

示例：
- 现象：`postgres` 数据源可创建但同步返回 `501`；
- 原因：`PostgresConnector` 仍为 phase 2 占位实现；
- 结论：文档中需标记为“暂未支持，不作为本期验收失败”。

## 5. 依赖与风险点

### 5.1 内容依赖
- 依赖现有代码结构稳定：特别是 `app/api/`、`app/schemas/`、`app/services/` 的接口行为；
- 依赖现有说明文档可复用：`README.md` 与 `docs/usage.md`；
- 依赖后续编写阶段不引入未经验证的示例能力。

### 5.2 主要风险

#### 风险一：场景文档与参考文档内容重复，后续容易漂移
应对策略：
- 把 `docs/use-cases.md` 定位为“场景化入口”；
- 把 `docs/usage.md` 定位为“配置与接口参考”；
- 案例中只保留必要请求示例，详细字段解释尽量引用或复用参考文档表述。

#### 风险二：误把预留能力写成已实现能力
高风险项：
- `postgres` 数据源同步；
- Redis 队列异步消费；
- MinIO 原文归档；
- 真正向量索引与生产级依赖联通；
- LLM 问答生成。

应对策略：
- 所有能力点以代码现状为准；
- 对“可创建但不可完整使用”的能力加醒目标识。

#### 风险三：案例太偏开发视角，导致“像接口文档，不像使用手册”
应对策略：
- 每个案例必须先写“场景目标”；
- 将代码细节下沉到“注意事项”或“补充说明”；
- 控制同一章节中长篇字段表与大段 JSON 的比例。

#### 风险四：缺少真实示例数据，影响可操作性
应对策略：
- 优先使用仓库已有字段模型构造最小可运行示例；
- 对未提供样例文件/Mock API 的部分，写清“示例数据要求”，不伪造仓库已有资产。

### 5.3 一期已知限制
下列内容需在后续最终文档中明确标注：
- 仓储默认为内存实现，服务重启会丢失数据；
- 健康检查更多是配置级检查，不代表依赖真实连通；
- `SYNC_RUN_INLINE=false` 时无后台消费机制；
- `postgres` 连接器尚未实现；
- 问答为检索结果拼接，不是大模型推理；
- 哈希向量仅用于开发验证，非生产检索质量承诺。

## 6. 工作量估算

### 6.1 工作拆解建议

| 工作项 | 说明 | 预估 |
| --- | --- | --- |
| 文档结构设计 | 确定章节、案例模板、信息分层 | 0.5 人日 |
| 代码与现有文档核对 | 核验接口、字段、边界与示例 | 0.5 人日 |
| 主流程案例编写 | 6 个主流程案例 | 1.0 人日 |
| 异常与边界章节编写 | 6 个边界场景 + FAQ | 0.5 人日 |
| 一致性校验与润色 | 校验字段、枚举、路径、语言表达 | 0.5 人日 |

### 6.2 总体估算
- 合计约 **3.0 人日**；
- 若仅产出面向研发/联调的精简版，可压缩到 **1.5 ~ 2.0 人日**；
- 若需要补充真实样例数据或截图，工作量会进一步上升。

### 6.3 完成标准
本方案对应的后续实现完成时，应满足：
- 新文档具备清晰章节结构；
- 至少覆盖 6 个主流程案例；
- 边界与限制表达完整且与代码一致；
- 读者无需先理解代码，即可按文档完成基础联调。

## 7. 结论
本次“技术方案”不是为系统新增功能设计技术实现，而是为“使用案例/使用手册”这一文档交付物建立可执行的内容架构、事实来源、案例模板与校验规则。

后续最优落地方式是：
- 以新增 `docs/use-cases.md` 为主交付物；
- 以 `docs/usage.md` 作为接口参考底座；
- 严格基于当前仓库代码现状编写，不扩写未实现能力。

该方案可以在不影响现有说明文档可读性的前提下，补足项目“首次上手难、场景说明弱、边界说明不够显式”的问题。
