# 使用案例编写 - 需求分析

## 1. 需求背景与目标

### 1.1 背景
当前项目 `knowledge-indexer` 已具备较完整的一期骨架能力，仓库内已有 `README.md` 和 `docs/usage.md` 两类说明文档，覆盖了项目介绍、启动方式、接口示例、联调流程与限制说明。但从任务标题“使用案例编写”和任务描述“帮我编写一下这个项目的使用案例，类似使用手册”来看，当前需求并不只是保留接口说明，而是需要进一步沉淀一份更贴近“用户操作路径”和“场景化使用”的文档产物。

这类文档应当帮助接入方、开发、测试和联调人员快速理解：
- 这个项目适合解决什么问题；
- 在什么前置条件下可以使用；
- 典型场景下该按什么顺序操作；
- 每一步应该看到什么结果；
- 当前版本有哪些边界和限制。

### 1.2 目标
本次需求的核心目标是：基于现有实现与已有文档，整理并输出一份“类似使用手册”的项目使用案例文档，重点突出“场景化说明 + 操作步骤 + 预期结果 + 注意事项”。

预期该文档应达到以下效果：
- 让首次接触项目的同学可在较短时间内完成本地启动与基础体验；
- 让接入方明确如何创建数据源、触发同步、执行检索与问答验证；
- 让联调/测试同学了解常见成功路径与失败路径；
- 让阅读者明确一期能力范围与未实现边界，避免错误预期。

### 1.3 交付目标
结合任务语义，本次后续文档产出应更偏“使用案例说明文档”，而不是纯粹的 API Reference。建议最终交付内容至少包括：
- 项目简介与适用场景；
- 角色视角的典型使用案例；
- 标准操作流程（启动、配置、创建数据源、同步、搜索、问答）；
- 异常/边界场景说明；
- 常见问题与注意事项。

## 2. 目标用户

结合现有仓库说明与接口设计，目标读者主要包括以下几类：

### 2.1 接入开发同学
负责从 Java 后端或其他内部服务接入 `knowledge-indexer`，关注：
- 内部接口如何调用；
- 鉴权如何配置；
- 搜索/问答请求结构；
- ACL 上下文如何传递；
- 返回结果如何消费。

### 2.2 本地开发与联调同学
负责本地启动、演示和排查问题，关注：
- 环境准备；
- `.env` 配置；
- 如何创建测试数据源；
- 如何验证索引、检索和问答链路是否跑通；
- 常见报错与定位方式。

### 2.3 测试/验收同学
负责验证一期骨架能力是否可用，关注：
- 支持哪些能力；
- 哪些是占位能力；
- 正常流程和异常流程的预期结果；
- 哪些问题属于已知限制而非缺陷。

### 2.4 项目负责人/产品相关方
希望快速理解系统价值与使用边界，关注：
- 项目定位；
- 当前可演示能力；
- 不适用场景；
- 后续演进方向。

## 3. 功能清单（主流程 + 边界场景）

本需求中的“功能清单”指最终使用案例/使用手册需要覆盖的内容范围。

### 3.1 主流程场景

#### 场景一：本地快速启动并确认服务可用
目的：帮助首次使用者完成环境准备并确认服务启动成功。
应覆盖内容：
- Python 版本要求与虚拟环境创建；
- 依赖安装与环境变量配置；
- 启动 `uvicorn` 服务；
- 访问 `/`、`/health`、`/docs` 验证服务状态；
- 如何判断当前运行在“一期本地验证模式”。

#### 场景二：创建文件数据源并完成一次全量同步
目的：帮助用户快速用本地 Markdown/TXT 文件体验索引能力。
应覆盖内容：
- `file` 类型数据源的配置项含义；
- `root_path` 与 `file_patterns` 的配置方式；
- 调用 `/internal/sources` 创建数据源；
- 调用 `/internal/sources/{id}/sync` 触发 `full` 同步；
- 调用 `/internal/jobs/{id}` 查看任务结果；
- 成功后可用于搜索/问答验证。

#### 场景三：创建 API 数据源并完成一次增量同步
目的：说明如何把外部知识接口接入本系统。
应覆盖内容：
- `api` 类型数据源的配置方式；
- API 返回格式要求（数组或包含 `items` 的对象）；
- 增量同步时 `checkpoint` 参数机制；
- 触发 `incremental` 同步后的预期行为；
- API 文档字段与 ACL 数据的映射关系。

#### 场景四：执行检索验证
目的：验证索引是否有效，以及过滤条件是否生效。
应覆盖内容：
- `/internal/search` 的请求结构；
- `query`、`top_k`、`filters`、`acl_context` 的作用；
- 检索结果中的 `score`、`source`、`document`、`citation` 含义；
- 典型查询样例；
- 如何通过过滤条件缩小结果范围。

#### 场景五：执行问答验证
目的：验证问答接口如何基于检索结果返回回答。
应覆盖内容：
- `/internal/ask` 的请求方式；
- 问答结果中 `answer`、`citations`、`evidence_status`、`reason` 的含义；
- 命中充分时的返回表现；
- 证据不足时的兜底表现；
- 当前问答并非大模型生成，而是检索结果拼接。

#### 场景六：带权限上下文的搜索/问答联调
目的：帮助接入方理解 ACL 的使用方式。
应覆盖内容：
- `user`、`role`、`department`、`tag` 四类 ACL；
- `allow` / `deny` 的优先关系；
- Java 后端向本服务传递权限上下文的方式；
- 权限命中与未命中的结果差异。

### 3.2 边界与异常场景

#### 边界一：未配置或错误配置内部鉴权
应覆盖内容：
- `INTERNAL_API_TOKEN` 为空时的行为；
- 已配置 Token 但请求未带/带错 `X-Internal-Token` 时的表现；
- `/health` 与 `/internal/*` 鉴权差异。

#### 边界二：同步任务未真正执行
应覆盖内容：
- `SYNC_RUN_INLINE=false` 时仅创建任务、不自动消费的行为；
- 当前版本未接入后台任务队列；
- 用户如何识别这是“当前设计限制”而非接口异常。

#### 边界三：重启服务后数据丢失
应覆盖内容：
- 当前默认基于内存仓储；
- 数据源、任务、索引结果不持久化；
- 这对演示、联调和验收的影响。

#### 边界四：`postgres` 数据源仅为占位
应覆盖内容：
- 可以创建但无法完成同步；
- 同步时会返回未实现错误；
- 文档中需要明确标注为“暂未支持”。

#### 边界五：问答返回“证据不足”
应覆盖内容：
- 命中文档数量不足；
- 检索分数低于阈值；
- 用户可如何调整查询词、过滤条件或测试数据。

#### 边界六：健康检查并不代表依赖已真正连通
应覆盖内容：
- `database`、`redis`、`minio` 当前更偏配置级检查；
- 不能把健康检查中的“已配置”直接理解为“已可用于生产”。

## 4. 验收标准

本步骤面向后续“使用案例文档编写”提供范围定义，后续交付物建议按以下标准验收：

### 4.1 内容完整性
- 文档需覆盖项目定位、适用对象、启动方式、鉴权、数据源管理、同步、检索、问答、ACL、常见问题与限制；
- 文档需至少包含 3 个以上典型使用案例，且案例具备“场景目标、操作步骤、预期结果”；
- 文档需明确区分“当前已实现能力”与“预留/未实现能力”。

### 4.2 可操作性
- 每个主流程案例都应能让读者按照步骤实际操作；
- 关键步骤需给出命令、接口调用方式或请求示例；
- 每个案例需说明成功判定标准，例如任务状态、返回字段或页面访问结果。

### 4.3 可理解性
- 文档语言应面向使用者，而非仅面向开发者；
- 章节结构应清晰，支持首次阅读者快速定位；
- 应减少纯代码导向描述，增强“做什么、为什么、结果是什么”的说明。

### 4.4 边界清晰性
- 必须明确标注以下一期限制：内存仓储、无异步任务队列、`postgres` 同步未实现、当前问答为轻量拼接逻辑；
- 对常见失败场景需给出解释，避免使用者误判为缺陷。

### 4.5 与现有实现一致
- 文档中出现的接口、字段、环境变量、同步模式、返回结构需与当前代码和现有文档一致；
- 不得引入当前仓库中不存在的能力描述。

## 5. 约束与边界

### 5.1 范围内
- 基于当前仓库已有代码与文档编写一期使用案例；
- 使用当前已暴露的内部接口与脚本作为说明基础；
- 强调本地验证、内部联调、能力演示等实际场景。

### 5.2 范围外
- 不扩展为正式对外产品说明书；
- 不虚构二期能力为当前已可用能力；
- 不把 PostgreSQL、Redis、MinIO、CocoIndex 的预留接口写成完整可操作方案；
- 不把当前问答能力描述为真实 LLM 问答系统。

## 6. 疑问点

当前基于仓库信息已能完成需求分析，但仍有以下待确认项，后续在技术方案或正式写作阶段可进一步确认：
- 最终交付文档希望落在 `docs/usage.md` 上直接增强，还是新增独立的“使用案例/使用手册”文档；
- 文档主要面向“研发/联调”还是也需要兼顾“非开发角色”的阅读体验；
- 是否需要加入更强的业务案例表达，例如“HR 知识库检索”“制度问答”等行业化示例；
- 是否需要保留现有 API 说明型写法，还是改造成更偏操作手册的章节结构；
- 是否需要配合补充示例数据目录或样例请求集合，以提升文档可复用性。

## 7. 结论

本次需求本质上不是新增系统功能，而是围绕现有 `knowledge-indexer` 一期能力，整理一份更贴近“使用手册”的场景化文档。后续工作应围绕“谁来用、怎么用、在哪些场景下用、成功与失败时分别会看到什么”展开，而不是重复已有的底层实现说明。
