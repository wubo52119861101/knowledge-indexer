## 2. 子任务详情与完成标准

### T01 手册信息架构与案例模板定稿
- 目标：把现有“接口说明型”文档重构为“使用手册型”信息架构，确定最终案例模板与章节顺序。
- 关键实现要点：
  - 基于 `requirements.md` 的 6 个主流程场景和 6 个边界场景设计目录；
  - 明确“项目简介 → 角色视角 → 前置条件 → 操作步骤 → 预期结果 → 注意事项”的统一结构；
  - 确定现有 `README.md` 与 `docs/usage.md` 的分工边界，避免重复堆叠接口字段表。
- 完成标准（DoD）：
  - 输出统一章节结构与案例模板；
  - 明确主文档落点与 README 的入口职责；
  - 后续子任务可直接按模板填充内容，无需再次重构目录。

### T02 编写项目概览与快速上手章节
- 目标：让首次接触项目的读者能快速理解项目定位、适用人群、启动方式和验证路径。
- 关键实现要点：
  - 提炼项目定位、当前能力边界和适用场景；
  - 编写环境要求、依赖安装、`.env` 配置、`uvicorn` 启动和 `/`、`/health`、`/docs` 验证步骤；
  - 补足“一期本地验证模式”的判断标准与注意事项；
  - 如有必要，同步精简 `README.md`，使其承担导航入口而非全文手册职责。
- 完成标准（DoD）：
  - 新读者可仅按文档完成本地启动；
  - 快速上手章节包含前置条件、操作步骤、预期结果；
  - `README.md` 到手册主文档的入口清晰可见。

### T03 编写数据源接入与同步案例
- 目标：覆盖文件源、API 源、同步任务和 `postgres` 占位场景，建立“创建数据源 → 触发同步 → 查看任务”的完整链路。
- 关键实现要点：
  - 编写 `file` 数据源创建与 `full` 同步案例；
  - 编写 `api` 数据源创建、返回格式要求与 `incremental` 同步案例；
  - 写清 `/internal/sources`、`/internal/sources/{id}/sync`、`/internal/jobs/{id}` 的请求示例和预期结果；
  - 标注 `postgres` 数据源“可创建、不可同步”的一期边界；
  - 说明 `SYNC_RUN_INLINE` 对任务执行行为的影响。
- 完成标准（DoD）：
  - 至少覆盖 2 个可操作主流程案例和 2 个同步相关边界说明；
  - 所有示例字段与接口路径和代码实现一致；
  - 读者能据此完成一次成功同步，并识别未执行/未实现场景。

### T04 编写检索、问答与 ACL 联调案例
- 目标：覆盖搜索验证、问答验证及权限上下文联调，让接入方理解结果含义与过滤行为。
- 关键实现要点：
  - 编写 `/internal/search` 的典型请求、结果字段和过滤样例；
  - 编写 `/internal/ask` 的命中充分与证据不足两类返回案例；
  - 解释 `query`、`top_k`、`filters`、`acl_context` 与 `citations`、`evidence_status`、`reason` 的业务含义；
  - 补充 `user`、`role`、`department`、`tag` 四类 ACL 及 `allow/deny` 的联调说明；
  - 明确当前问答是检索拼接结果，而非大模型自由生成。
- 完成标准（DoD）：
  - 至少覆盖 3 个主流程案例：搜索、问答、ACL 联调；
  - 检索与问答示例能体现正常命中和边界兜底差异；
  - ACL 行为说明与代码判定逻辑一致，不引入虚假能力。

### T05 编写异常边界、限制与 FAQ
- 目标：系统化沉淀异常场景、已知限制和常见问题，避免使用方产生错误预期。
- 关键实现要点：
  - 覆盖鉴权错误、任务未执行、服务重启丢数据、`postgres` 未实现、问答证据不足、健康检查仅配置级验证等场景；
  - 每个场景按“现象 → 原因 → 结论/建议”结构编写；
  - 将一期限制与真正缺陷区分开，避免误判；
  - 适当补充 FAQ，承接主流程中的常见困惑。
- 完成标准（DoD）：
  - 至少覆盖 6 个边界/异常场景；
  - 每个场景都给出解释与使用建议；
  - 文档明确标注哪些属于一期设计限制。

### T06 文档一致性校验与入口联动
- 目标：在交付前完成事实核对、语言润色和入口联动，保证手册可读、可用、可维护。
- 关键实现要点：
  - 核对接口路径、字段名、枚举值、环境变量、脚本名、路径引用与当前代码一致；
  - 消除 `README.md` 与 `docs/usage.md` 的重复与冲突；
  - 补充目录跳转、相关推荐入口和必要提示；
  - 按“使用手册”风格统一语气，减少纯 API Reference 口吻。
- 完成标准（DoD）：
  - 关键事实项完成逐项核对，无明显代码漂移；
  - 读者从 `README.md` 能自然进入主手册；
  - 文档整体结构清晰，适合研发、联调和测试共同使用。

## 3. 优先级排序

1. `T01` 手册信息架构与案例模板定稿
2. `T02` 编写项目概览与快速上手章节
3. `T03` 编写数据源接入与同步案例
4. `T04` 编写检索、问答与 ACL 联调案例
5. `T05` 编写异常边界、限制与 FAQ
6. `T06` 文档一致性校验与入口联动

说明：`T05` 的业务优先级略低于主流程案例，但必须在交付前完成；`T06` 虽然工作量小，但属于最终交付的阻塞项，因此整体优先级保持 P0。

## 4. 并行 vs 串行执行建议

### 建议串行执行
- `T01 → T02`：先定结构，再写导读与快速上手，避免返工；
- `T03 → T04`：检索/问答案例应承接前文的数据源与同步上下文；
- `T05 → T06`：边界说明收敛后再做一致性校验与入口联动。

### 可有限并行执行
- `T02` 与 `T03`：在 `T01` 完成后，可由不同人员分别编写“快速上手”和“数据源同步”章节；
- `T04` 可在 `T03` 中后期并行起草，但最终需要按案例顺序统一串接；
- `T05` 可在主流程案例基本成形后提前整理草稿，待主文稳定后统一收口。

## 5. 整体完成标准

- 至少覆盖 `requirements.md` 中定义的 6 个主流程场景与 6 个边界场景；
- 主文档具备统一案例模板，读者无需先读代码即可完成基本联调；
- 所有事实性内容与当前仓库实现一致，不把预留能力误写为已实现能力；
- `README.md` 与主手册职责清晰，项目入口与详细说明互相可达。
