# 需求分析

## 1. 需求背景与目标

### 背景
当前 `knowledge-indexer` 已具备企业知识库一期的基础能力，包括数据源接入、同步建索引、内部检索接口 `/internal/search`、内部问答接口 `/internal/ask`、任务查询接口 `/internal/jobs/{id}` 等。现有版本以“先跑通链路”为目标，整体实现偏向骨架能力，仍存在以下明显限制：

1. `/internal/ask` 当前并非完整的大模型生成链路，而是将检索结果片段直接拼装成回答。
2. 任务系统缺少取消任务的 HTTP 接口，`CANCELLED` 仅作为兼容性预留能力，无法被外部显式触发。
3. `rerank` 仅为扩展点，默认未启用，问答与检索结果排序仍主要依赖当前召回顺序与分数。
4. `pipeline_engine` 对外展示为内置索引引擎，无法准确体现外部编排引擎的接入状态或执行来源。

### 目标
本次迭代目标是围绕上述四项限制进行优化，在不破坏现有内部接口基本使用方式的前提下，提升系统的：

- **回答能力**：从“检索结果拼装”升级为“检索增强 + 大模型生成”的问答链路。
- **任务可运维性**：支持任务取消，提高同步任务控制能力。
- **排序扩展性**：让 `rerank` 成为可配置、可启用、可降级的正式能力。
- **执行链路可观测性**：准确表达当前请求/任务所使用的 `pipeline_engine`，避免对接方误判系统能力边界。

## 2. 目标用户

- **上游 Java 后端/业务服务**：通过内部接口接入检索、问答、同步任务能力。
- **联调与实施同学**：需要明确系统当前实际能力边界、任务状态和执行链路。
- **平台/算法/后端研发同学**：需要在后续迭代中继续替换模型、接入外部编排引擎或增强任务系统。

## 3. 功能清单

### 3.1 主流程

#### 功能一：问答链路升级为完整生成链路
用户通过 `/internal/ask` 发起问题后，系统应基于检索结果构建完整问答链路，而不是直接拼接命中文本。主流程应满足：

- 基于问题、过滤条件、ACL 上下文先执行检索。
- 在具备条件时支持引入 `rerank` 对候选结果进行二次排序。
- 基于最终证据构造提示词/上下文，调用大模型生成自然语言回答。
- 返回生成后的答案以及可追溯引用信息，继续保留“证据是否充分”的判断能力。
- 当证据不足、模型不可用或生成失败时，系统应回退到明确的兜底响应，而不是返回不可信内容。

#### 功能二：提供任务取消能力
针对同步任务，系统应提供可由外部调用的取消入口，支持对 `PENDING`/`RUNNING` 中的任务执行取消操作。主流程应满足：

- 调用方可通过 HTTP 接口发起取消。
- 任务状态流转需要体现取消中/已取消的结果，至少要能稳定落到 `CANCELLED` 语义。
- 取消操作应具备幂等性，重复取消不会造成状态异常。
- 任务被取消后，应停止后续不必要的处理，避免继续写入结果或继续消耗资源。

#### 功能三：将 rerank 从扩展点升级为正式能力
系统应保留默认关闭策略，但在配置满足时允许启用 `rerank`，并使其对检索/问答结果质量产生正向作用。主流程应满足：

- 默认行为与现状兼容，不启用时不影响现有调用方。
- 启用后，`rerank` 应参与问答证据排序，并影响最终返回答案与引用顺序。
- 当 `rerank` 服务异常、超时或未配置时，可自动降级到原始检索排序。

#### 功能四：准确展示 pipeline_engine
系统对外暴露的 `pipeline_engine` 应准确反映当前实际使用的执行引擎，而不是固定显示为内置引擎。主流程应满足：

- 当使用内置索引/编排逻辑时，展示为内置引擎。
- 当已接入外部编排引擎时，展示为外部引擎及其对应标识。
- 对接方可通过接口返回或任务/健康信息准确判断当前执行来源。

### 3.2 边界场景

#### 问答链路边界
- 检索命中为空或证据数不足时，应返回“证据不足”而不是模型幻觉答案。
- 检索有结果但最高分过低时，应仍然判定为不可靠。
- 大模型调用超时、报错、限流时，应有明确降级策略与错误语义。
- 若启用了 `rerank`，但 `rerank` 异常，应自动回退为原始检索排序，不影响主流程可用性。
- ACL 过滤后无结果时，应与普通“证据不足”场景做一致处理，不能泄露不可见内容。

#### 任务取消边界
- 取消不存在的任务时，应返回明确错误。
- 已完成（`SUCCEEDED` / `FAILED`）任务不可取消，需返回可预期结果。
- 已取消任务重复取消时，应保证接口幂等。
- 对于正在执行的同步任务，取消生效点需要可定义；若无法立即中断，也应保证后续流程尽快停止。

#### pipeline_engine 边界
- 配置缺失、环境未启用外部编排引擎时，不能错误宣称已走外部引擎。
- 同一服务存在多种执行模式时，应能根据实际运行分支返回准确值，而非固定文案。

## 4. 验收标准

### 4.1 问答能力
- `/internal/ask` 在证据充分场景下返回的大部分内容来自大模型生成，而不是简单片段拼接。
- 返回结构中仍保留引用信息，调用方可定位答案依据。
- 在证据不足、模型失败、排序失败等异常场景下，系统能够稳定返回兜底结果。
- 现有 ACL、过滤条件、`top_k` 等调用方式保持兼容。

### 4.2 任务取消能力
- 系统提供可用的任务取消 HTTP 接口。
- 可成功取消 `PENDING` 或可取消的 `RUNNING` 任务。
- 取消后的任务状态可查询，且不会继续被当作成功任务处理。
- 重复取消、取消已结束任务、取消不存在任务等场景行为清晰且可预期。

### 4.3 rerank 能力
- `rerank` 默认关闭时，系统行为与当前版本保持兼容。
- 启用后，排序结果对问答回答和引用顺序产生影响。
- `rerank` 不可用时系统自动降级，不导致主链路不可用。

### 4.4 pipeline_engine 展示
- 对外展示的 `pipeline_engine` 与实际执行引擎一致。
- 接入方可通过接口或状态信息区分“内置引擎”和“外部编排引擎”。
- 文档中对 `pipeline_engine` 的含义、取值及判定逻辑有清晰说明。

### 4.5 文档与联调
- 使用文档需同步更新，明确新的问答链路、取消任务方式、`rerank` 行为和 `pipeline_engine` 说明。
- 核心场景至少覆盖：正常问答、证据不足问答、取消任务、`rerank` 启用/降级、执行引擎识别。

## 5. 非目标与边界约束

- 本次迭代目标是优化一期限制，不等同于一次性完成二期全部基础设施替换。
- 若未明确要求，本次不强制引入 PostgreSQL 持久化、Redis 任务队列、MinIO 原文归档等完整基础设施落地。
- 若外部编排引擎尚未完全接入，本次至少要保证 `pipeline_engine` 的表达真实、准确、可扩展。
- 需要优先保证现有内部 API 的基本兼容性，避免对上游调用方造成破坏性影响。

## 6. 疑问点

1. **大模型接入范围**：本次是只需要预留标准生成链路，还是必须接入指定模型/供应商并默认启用？
2. **问答结果兼容性**：是否要求 `/internal/ask` 维持当前响应结构完全不变，仅替换 `answer` 内容生成方式？
3. **取消语义深度**：对于已进入执行中的同步任务，是否要求“立即硬中断”，还是接受“在安全检查点尽快停止”的软取消？
4. **状态枚举约束**：`CANCELLED` 是否需要正式加入当前任务状态枚举及文档合同，还是仅作为接口返回值兼容扩展？
5. **rerank 配置方式**：启用开关是全局配置、按请求控制，还是两者都支持？
6. **pipeline_engine 暴露位置**：需要在哪些接口中体现该字段（如 `/health`、任务详情、问答/检索响应、数据源详情）？
7. **外部编排引擎定义**：当前目标中的“外部编排引擎”具体指 CocoIndex、其他 workflow engine，还是统一抽象层？

## 7. 后续双 Agent 协作建议（供下一步拆分参考）

- **Agent A：问答链路增强**
  - 聚焦 `/internal/ask`、检索链路、`rerank` 接入与降级策略。
  - 目标是完成“检索 → 可选 rerank → 生成 → 引用/兜底”的主链路需求。

- **Agent B：任务与执行链路治理**
  - 聚焦任务取消接口、任务状态机、`pipeline_engine` 对外表达与文档一致性。
  - 目标是提升任务可控性与执行可观测性。

> 说明：以上仅为后续协作建议，不替代正式的技术方案与任务拆分。
